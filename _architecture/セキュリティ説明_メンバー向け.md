# セキュリティ設計説明書（メンバー向け）

本ドキュメントは、CBLアプリのセキュリティ設計について、
社内メンバーが顧客や関係者からの質問に答えられるよう整理したものです。

---

## 1. 認証（Authentication）

### 使用サービス
**Supabase Auth**（基盤は GoTrue）

### 仕組み
- メールアドレス＋パスワードによる認証
- パスワードは **bcrypt** でハッシュ化して保存
  - ハッシュ化 = 元の文字列に戻せない一方向変換
  - データベースが漏洩しても、パスワード自体は解読不能
- ログイン成功後は **JWT（JSON Web Token）** をセッションとして発行
  - JWTはサーバーで署名されており、改ざん検知が可能
  - 有効期限あり（デフォルト1時間）、リフレッシュトークンで自動延長

## 2. データアクセス制御（Row Level Security）

### 使用機能
**Supabase RLS（Row Level Security）**= PostgreSQL の行レベルセキュリティ

### 仕組み
データベースのテーブルごとに「誰が何を読み書きできるか」をSQL で定義。
アプリのコードとは独立して、DB レベルで制御されるため、
コードにバグがあっても他人のデータが漏れない。

### 設定内容

#### profiles テーブル
```sql
-- 自分のプロフィールのみ読み書き可
create policy "自分のプロフィールのみ読み書き可"
  on profiles for all
  using (auth.uid() = id);

-- 管理者は全プロフィールを閲覧可（SELECT のみ）
create policy "管理者は全プロフィールを閲覧可"
  on profiles for select
  using (
    exists (
      select 1 from admin_users
      where email = auth.jwt() ->> 'email'
    )
  );
```

#### applications テーブル
```sql
-- 自分の申込のみ操作可
create policy "自分の申込のみ操作可"
  on applications for all
  using (auth.uid() = user_id);

-- 管理者は全申込を閲覧・更新可
create policy "管理者は全申込を閲覧可"
  on applications for select
  using (...);

create policy "管理者は申込ステータスを更新可"
  on applications for update
  using (...);
```

## 3. 通信の暗号化

- すべての通信は **HTTPS（TLS 1.2以上）** で暗号化
- Vercel（フロントエンド）と Supabase（バックエンド）の両方がデフォルトでHTTPS対応
- HTTP でのアクセスは自動的に HTTPS にリダイレクト

---

## 4. 管理者権限の制御

### 仕組み
- `admin_users` テーブルにメールアドレスを登録したユーザーのみ管理画面にアクセス可能
- RLS により、管理者でないユーザーが `/admin` を叩いても他人のデータは返らない
- フロント側のリダイレクトはUX用途、セキュリティの実態はRLSが担保

### 管理者追加方法
```sql
insert into admin_users (email) values ('new-admin@example.com');
```
SQL Editor または service_role キー経由でのみ追加可能（一般ユーザーは追加不可）。

## 5. インフラのセキュリティ

### Supabase
- **SOC 2 Type II** 認証取得済み
- データは暗号化して保存（AES-256）
- 東京リージョン（ap-northeast-1）対応
- 自動バックアップあり

### Vercel
- **ISO 27001** 認証取得済み
- エッジネットワーク経由で DDoS 対策済み
- 環境変数はビルド時のみ展開、ソースコードには含まれない

---

## 6. 環境変数の管理

```
NEXT_PUBLIC_SUPABASE_URL       # 公開可（APIエンドポイント）
NEXT_PUBLIC_SUPABASE_ANON_KEY  # 公開可（RLSで保護されているため）
```

- `ANON_KEY` は公開されても RLS があるため、他人のデータには触れない
- `SERVICE_ROLE_KEY`（管理者キー）は使用しておらず、フロントには含まれない
- `.env.local` は `.gitignore` で管理、GitHub には push しない

## まとめ

| 脅威 | 対策 |
|------|------|
| パスワード漏洩 | bcrypt ハッシュ化 |
| 他人のデータ閲覧 | RLS（DB レベルで制御） |
| 通信傍受 | HTTPS/TLS 暗号化 |
| 不正な管理者操作 | admin_users テーブル＋RLS |
| 環境変数漏洩 | .gitignore＋Vercel 環境変数管理 |
| インフラ攻撃 | Supabase/Vercel のセキュリティ基盤 |
